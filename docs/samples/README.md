## Ключевая структура реального XML-МЧД

```xml
<Доверенность xmlns="urn://x-artefacts/EMCHD_1" ВерсФорм="EMCHD_1" ИдФайл="...">
  <Документ>
    <Довер>
      <СвДов
        ВидДовер="1"
        НомДовер="uuid"
        ДатаВыдДовер="ГГГГ-ММ-ДД"
        СрокДейст="ГГГГ-ММ-ДД">
        <СведСист>https://m4d.nalog.gov.ru/emchd/check-status?guid=...</СведСист>
      </СвДов>

      <СвДоверит ТипДоверит="1|3">
        <!-- ТипДоверит=1: РосОргДовер → СвРосОрг (ИННЮЛ, КПП, ОГРН, НаимОрг)
             ТипДоверит=3: ИПДовер (ИННФЛ, ОГРНИП) -->
      </СвДоверит>

      <СвУпПред ТипПред="3">
        <Пред>
          <СведФизЛ ИННФЛ="..." СНИЛС="...">
            <СведФЛ ДатаРожд="...">
              <ФИО Фамилия="..." Имя="..." Отчество="..."/>
              <УдЛичнФЛ КодВидДок="21" СерНомДок="..." .../>
            </СведФЛ>
          </СведФизЛ>
        </Пред>
      </СвУпПред>

      <СвПолн ТипПолн="1">
        <МашПолн КодПолн="FAS_EIAS_MA0001"
                 НаимПолн="Описание полномочия"/>
        <!-- ...ещё полномочия -->
      </СвПолн>
    </Довер>
  </Документ>
</Доверенность>
```

## ⚠️ Важные отличия от учебного примера `mcd-example.xml`

Мой первый `mcd-example.xml` был написан по мотивам старого формата 5.03 и оказался **неточным**. Учитывайте при реализации парсера:

| Что | Мой учебный | Реальный формат |
|---|---|---|
| Корневой тег | `<Файл>` | `<Доверенность>` |
| Namespace | нет | `urn://x-artefacts/EMCHD_1` |
| Версия | `ВерсФорм="5.03"` | `ВерсФорм="EMCHD_1"` |
| Структура | `Документ/СвДов/СвДоверит/СвУпЛ/СвПолн` | `Документ/Довер/СвДов/СвДоверит/СвУпПред/СвПолн` |
| Представитель | `СвУпЛ/УпЛФЛ` | `СвУпПред/Пред/СведФизЛ` |
| Реквизиты ЮЛ | в отдельных тегах (`НаимОрг`, `ИННЮЛ`) | **атрибуты** тега `<СвРосОрг ИННЮЛ="..." ОГРН="..."/>` |
| Полномочия | `<Полн>` с числовым `02.08` (иерархия КодКласс/КодГруп/КодПодГруп/КодТип) | `<МашПолн КодПолн="FAS_EIAS_MA0001"/>` — **строковые ведомственные коды** |

**Главное:** в ЕКП коды полномочий — **не числовые** (`02.08`), а **строковые идентификаторы вида `{ВЕДОМСТВО}_{ПОДСИСТЕМА}_{КОД}`** (например `BBDOCS_DOCS_DCSALL_SRCDOC_SOURCEDOCS8`).

## Что это значит для нашей спеки

В [`docs/mcd-spec-v2.md`](../mcd-spec-v2.md) нужно обновить:
- Поле `McdPower.code` — оставить `string`, но **снять упоминание формата `02.08`** — будут реальные строковые коды
- `EKP_CATALOG` — не семёрка условных записей, а полноценный справочник с ведомственными кодами ФНС (несколько тысяч записей)
- `DOC_TYPE_REQUIRED_POWER` — сопоставить типы документов eTRN с реальными `КодПолн`:
  - `trn` → `BBDOCS_DOCS_DCSALL_SRCDOC_SOURCEDOCS8` (или `SOURCEDOCS7`, или `SOURCEDOCS5` — зависит от юридической трактовки)

Финальный справочник кодов и маппинг — задача бэкендера совместно с юристом / методологом ЭДО.

## Как получить свой тестовый XML

1. Зайти на https://m4d.nalog.gov.ru/emchd/create через ЭП руководителя
2. Заполнить форму (доверитель / представитель / полномочия)
3. Скачать XML перед отправкой в реестр — это и будет ваш тестовый файл
